**Проект:** Регулятор безопасности, DevOps, DevSecOps\
**Основание:** ТЗ С8 «Регулятор», ТЗ П1 «DevOps», ТЗ П2 «DevSecOps»\
**Цель:** проверить соответствие системы функциональным, интеграционным,
требованиям безопасности, требованиям к CI/CD и SAST

**Цель тестирования**

Цель тестирования --- подтвердить, что система:

-   реализует функции регулятора безопасности;

-   обеспечивает обмен между доменами безопасности только через брокер
    сообщений;

-   проверяет политики безопасности через монитор безопасности;

-   корректно обрабатывает сертификаты безопасности;

-   передаёт телеметрию и события безопасности;

-   соответствует требованиям DevOps по развёртыванию, автоматическим
    проверкам и работе в чистом окружении;

-   соответствует требованиям DevSecOps по проверке зависимостей и SAST.

**Объект тестирования**

Объектами тестирования являются следующие компоненты системы:

-   регулятор безопасности;

-   механизм генерации и проверки сертификатов;

-   механизм отзыва сертификатов;

-   механизм проверки покрытия кода;

-   брокер сообщений;

-   монитор безопасности;

-   сервис аналитики;

-   подсистема логирования;

-   CI/CD pipeline;

-   SAST-инструменты;

-   скрипты развёртывания и настройки окружения.

## Что входит в объём тестирования

В рамках тестирования проверяются:

-   генерация сертификата с заданными целями безопасности;

-   проверка подлинности сертификата;

-   отзыв сертификата;

-   передача телеметрии в наземные системы;

-   передача событий безопасности в сервис аналитики;

-   проверка соответствия политик безопасности;

-   контроль покрытия тестами;

-   выполнение автотестов в CI/CD;

-   работа pre-commit hook;

-   анализ зависимостей на наличие уязвимостей;

-   запуск SAST и блокировка изменений при ошибках;

-   логирование событий с указанием уровня критичности.

## Что не входит в объём тестирования

Не тестируются отдельно:

-   физическая инфраструктура, если она не влияет на работу приложения;

-   сторонние системы, если они не участвуют в сценариях интеграции;

-   ручная эксплуатация вне сценариев, описанных в ТЗ;

-   неописанные в ТЗ бизнес-функции.

## Подход к тестированию

Тестирование выполняется на нескольких уровнях:

### 6.1 Unit-тестирование

Проверяется логика:

-   генерации сертификата;

-   проверки подписи;

-   расчёта покрытия;

-   обработки ошибок;

-   формирования событий и логов.

### 6.2 Integration-тестирование

Проверяется:

-   взаимодействие с брокером;

-   взаимодействие с монитором безопасности;

-   передача данных в сервис аналитики;

-   работа с CI/CD;

-   интеграция SAST и проверки зависимостей.

### 6.3 System-тестирование

Проверяется система в целом:

-   полный цикл генерации и проверки сертификата;

-   отзыв сертификата;

-   передача телеметрии;

-   обработка событий безопасности;

-   соблюдение политики безопасности.

### 6.4 Acceptance-тестирование

Проверяется соответствие системы ТЗ:

-   все обязательные требования выполнены;

-   покрытие выдержано;

-   безопасность не нарушена;

-   CI/CD и SAST работают автоматически.

## Типы тестирования

### Функциональное

-   генерация сертификатов;

-   проверка сертификатов;

-   отзыв сертификатов;

-   запрос и выдача целей безопасности;

-   запрос на выплату страховки.

### Интеграционное

-   взаимодействие через брокер сообщений;

-   передача телеметрии;

-   передача событий безопасности;

-   взаимодействие с монитором безопасности;

-   работа с сервисом аналитики.

### Безопасность

-   запрет обхода брокера;

-   проверка политик безопасности;

-   проверка уровня доверия доменов;

-   анализ зависимостей на уязвимости;

-   SAST;

-   проверка обработки небезопасных входных данных.

-   проверка логирования инцидентов безопасности;

### DevOps

-   развёртывание на чистой среде;

-   выполнение автотестов;

-   pre-commit hook;

-   запуск pipeline.

### Требования к CI/CD pipeline

Pipeline должен включать этапы:

-   сборка (build);

-   запуск unit-тестов;

-   проверка покрытия кода;

-   анализ зависимостей;

-   запуск SAST;

-   развёртывание (deploy).

Pipeline считается неуспешным, если:

-   unit-тесты завершились с ошибкой;

-   покрытие кода ниже установленного порога;

-   обнаружены критические уязвимости (SAST);

-   обнаружены критические уязвимости в зависимостях.

> При неуспешном pipeline:

-   изменения не допускаются к слиянию;

-   разработчик получает отчёт об ошибках.

### DevSecOps

-   проверка зависимостей;

-   запуск SAST;

-   блокировка commit/push при ошибках;

-   интеграция в CI/CD.

## Негативные сценарии

Проверяются некорректные и атакующие сценарии:

-   генерация сертификата с некорректными параметрами;

-   использование сертификата без подписи;

-   использование просроченного сертификата;

-   использование отозванного сертификата;

-   попытка подмены сертификата;

-   попытка обхода брокера сообщений (прямые вызовы API);

-   отправка сообщений, нарушающих политики безопасности;

-   внедрение вредоносных входных данных (инъекции, некорректные
    форматы);

-   повторная отправка (replay) событий безопасности;

-   подмена телеметрии;

-   использование зависимостей с известными критическими уязвимостями;

-   наличие секретов (ключей, токенов) в коде.

Ожидаемый результат:

-   система отклоняет некорректные данные;

-   фиксируются события безопасности;

-   нарушения регистрируются монитором безопасности;

-   выполнение операций блокируется.

## Критерии начала тестирования

Тестирование начинается, если:

-   доступен исходный код;

-   настроено тестовое окружение;

-   есть инструкции по запуску;

-   доступны автотесты;

-   доступны скрипты развёртывания;

-   известны требования ТЗ.

## Критерии завершения тестирования

Тестирование считается завершённым, если:

-   выполнены все запланированные тесты;

-   пройдены все обязательные проверки;

-   нет критических дефектов;

-   покрытие кода соответствует требованиям;

-   SAST не выявляет критических уязвимостей;

-   зависимости проверены;

-   CI/CD и pre-commit работают корректно;

-   сформированы отчёты по тестам и покрытию.

## Метрики качества

В процессе тестирования контролируются следующие метрики:

-   процент успешного прохождения тестов;

-   уровень покрытия кода (%);

-   количество дефектов по уровням критичности;

-   количество найденных уязвимостей;

-   время выполнения CI/CD pipeline;

-   количество заблокированных commit/push.

Метрики используются для оценки качества системы и готовности к приёмке.

## Основные тестовые сценарии

### TC-01. Генерация сертификата

**Цель:** проверить обработку запроса на генерацию сертификата.\
**Шаги:**

1.  Отправить запрос с целями безопасности.

2.  Указать корректные параметры.

3.  Выполнить генерацию.

**Ожидаемый результат:**

-   сертификат создан;

-   в нём указаны цели безопасности;

-   сертификат подписан Регулятором или АСЦ.

### TC-02. Проверка подлинности сертификата

Цель: проверить валидацию сертификата.

Входные данные:

-   валидный сертификат (подписан);

-   сертификат без подписи;

-   поддельный сертификат;

-   отозванный сертификат;

-   просроченный сертификат.

Шаги:

1.  Передать каждый тип сертификата в систему.

2.  Выполнить проверку.

Ожидаемый результат:

-   валидный сертификат подтверждается;

-   невалидные сертификаты отклоняются;

-   причины отказа логируются;

-   событие фиксируется системой безопасности.

### TC-03. Отзыв сертификата

**Цель:** проверить отзыв сертификата.\
**Шаги:**

1.  Выбрать выданный сертификат.

2.  Выполнить отзыв.

3.  Повторно проверить сертификат.

**Ожидаемый результат:**

-   сертификат помечается как отозванный;

-   повторная проверка возвращает отказ.

### TC-04. Передача телеметрии

**Цель:** проверить передачу телеметрии в наземные системы.\
**Шаги:**

1.  Сгенерировать телеметрию.

2.  Передать данные в сервис аналитики.

**Ожидаемый результат:**

-   телеметрия передана;

-   событие зафиксировано в логах.

### TC-05. Передача событий безопасности

**Цель:** проверить передачу событий безопасности в сервис аналитики.\
**Шаги:**

1.  Сформировать событие безопасности.

2.  Передать его в сервис аналитики.

**Ожидаемый результат:**

-   событие успешно доставлено;

-   сохранён уровень критичности.

### TC-06. Проверка взаимодействия только через брокер

**Цель:** проверить выполнение БТ2 ТЗ С8.\
**Шаги:**

1.  Попытаться организовать прямое взаимодействие между доменами.

2.  Сравнить поведение с передачей через брокер.

**Ожидаемый результат:**

-   прямое взаимодействие запрещено;

-   обмен проходит только через брокер;

-   монитор фиксирует нарушение, если оно было попыткой.

### TC-07. Проверка политик безопасности

**Цель:** проверить автоматизированную проверку политик.\
**Шаги:**

1.  Передать сообщение, нарушающее политику.

2.  Передать сообщение, соответствующее политике.

**Ожидаемый результат:**

-   корректное сообщение проходит;

-   некорректное отклоняется.

### TC-08. Проверка покрытия кода

**Цель:** проверить выполнение БТ5, БТ6, БТ7.\
**Шаги:**

1.  Запустить отчёт покрытия.

2.  Сравнить результат с порогом.

**Ожидаемый результат:**

-   доверенный код ≥ 60%;

-   критичный доверенный код ≥ 70% с учётом зависимостей;

-   недоверенный код ≥ 30%.

### TC-09. Проверка зависимостей

**Цель:** проверить наличие уязвимостей в зависимостях.\
**Шаги:**

1.  Запустить анализ зависимостей.

2.  Проверить отчёт.

**Ожидаемый результат:**

-   критические уязвимости отсутствуют;

-   уязвимые пакеты не используются в доверенных доменах.

### TC-10. Проверка SAST

**Цель:** проверить запуск статического анализа.\
**Шаги:**

1.  Запустить SAST на коде.

2.  Проверить формирование отчёта.

3.  Проверить реакцию pipeline на критические ошибки.

**Ожидаемый результат:**

-   SAST выполнен;

-   отчёт сформирован;

-   при критических уязвимостях pipeline блокируется.

### TC-11. Проверка pre-commit hook

**Цель:** проверить автоматические проверки перед отправкой изменений.\
**Шаги:**

1.  Внести изменение в код.

2.  Попробовать выполнить commit/push с ошибкой тестов или SAST.

**Ожидаемый результат:**

-   проверка запускается автоматически;

-   при ошибке отправка изменений блокируется.

### TC-12. Проверка развёртывания на чистой среде

**Цель:** проверить выполнение требований DevOps.\
**Шаги:**

1.  Развернуть систему на чистой Ubuntu-среде или в Docker.

2.  Запустить все автотесты.

**Ожидаемый результат:**

-   развёртывание проходит без ручных исправлений;

-   все автотесты завершаются успешно.

### TC-13. Проверка невалидного сертификата

Цель: убедиться, что система отклоняет некорректные сертификаты.

Шаги:

1.  Передать сертификат без подписи.

2.  Передать просроченный сертификат.

3.  Передать отозванный сертификат.

**Ожидаемый результат:**\
• все сертификаты отклоняются;\
• возвращается ошибка валидации;\
• событие фиксируется в системе логирования.

### TC-14. Попытка обхода брокера сообщений

Цель: проверить запрет прямого взаимодействия между доменами.

Шаги:

1.  Попытаться отправить сообщение напрямую между сервисами (минуя
    брокер).

2.  Выполнить тот же сценарий через брокер.

**Ожидаемый результат:**\
• прямое взаимодействие блокируется;\
• через брокер сообщение проходит;\
• монитор безопасности фиксирует попытку нарушения.

### TC-15. Проверка уязвимых зависимостей

Цель: убедиться, что система обнаруживает уязвимые зависимости.

Шаги:

1.  Добавить зависимость с известной критической уязвимостью.

2.  Запустить анализ зависимостей.

Ожидаемый результат:\
• уязвимость обнаружена;\
• формируется отчёт;\
• использование зависимости запрещено в доверенном домене.

### TC-16. Блокировка pipeline при ошибках SAST

Цель: проверить реакцию CI/CD на критические уязвимости.

Шаги:

1.  Добавить в код уязвимость.

2.  Запустить pipeline.

**Ожидаемый результат:**\
• SAST обнаруживает проблему;\
• pipeline завершается с ошибкой;\
• изменения не допускаются к слиянию.

## Критерии приёмки

Система считается соответствующей ТЗ, если:

-   все обязательные функции регулятора реализованы;

-   сертификаты создаются, проверяются и отзываются;

-   взаимодействие между доменами идёт только через брокер;

-   безопасность контролируется монитором;

-   телеметрия и события передаются корректно;

-   события логируются с указанием критичности;

-   покрытие соответствует требованиям;

-   зависимости проверены на уязвимости;

-   SAST и pre-commit работают;

-   CI/CD функционирует;

-   развёртывание в чистом окружении успешно.

## Риски

-   неполное или неоднозначное ТЗ;

-   отсутствие стабильной тестовой среды;

-   сложности с проверкой security policy;

-   наличие уязвимых зависимостей;

-   отсутствие реальных данных для интеграционных тестов;

-   различие между учебной и боевой реализацией сертификатов;

-   недоступность внешних SAST-инструментов или брокера.

-   test plan;

-   test cases;

-   отчёт о покрытии кода;

-   отчёт SAST;

-   отчёт по проверке зависимостей;

-   список найденных дефектов;

-   отчёт о прохождении CI/CD;

-   итоговый отчёт о соответствии ТЗ.
